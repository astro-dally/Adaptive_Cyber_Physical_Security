Zero-Day Intrusion Detection for Network Traffic

This project studies how to detect both known attacks and unseen attacks in CIC-IDS-2018 traffic. The core design choice is deliberate: instead of relying on a single model, the system combines a supervised classifier for familiar attack behavior with an anomaly detector for novelty.

Core Question

01 / Problem Framing

Why This Project Exists

Security Problem

Traditional intrusion detection often assumes the future looks like the training set. That assumption is weak in practice because adversaries change tactics.

Research Goal

Build a workflow that supports both closed-world classification and open-world anomaly detection on the same network-flow dataset.

Design Choice

Use Random Forest for known threats and One-Class SVM for novelty detection, then compare the roles each model plays in a realistic zero-day setting.

Decision: the project is not trying to claim a production-ready IDS. It is building a rigorous Phase-1 foundation with evidence, justification, and reproducible artifacts.

04 / Pipeline Design

From Raw Network Flows to Modeling-Ready Features

1

Schema Cleanup

Drop flow identifiers and endpoint metadata such as Flow ID, source IP, destination IP, and timestamps because they are either non-generalizable or leakage-prone.

2

Numeric Normalization

Convert non-label columns to numeric form, replace infinite values, and handle missingness so the data is stable for both tree models and kernel methods.

3

Information Filtering

Remove constant columns and strip the stray label artifact before building BinaryLabel and AttackType.

4

Redundancy Reduction

Drop highly correlated features above 0.98 using simple domain heuristics so the retained space is more compact and less repetitive.

After Feature Engineering

The engineered space is cleaner and more consistent with downstream model assumptions.

Reduced Correlation Structure

This is the point of the feature engineering stage: preserve signal while cutting obvious redundancy.

05 / Theoretical Rigor

Why These Models Make Sense

Random Forest

Random Forest aggregates many decision trees built on bootstrap samples and random feature subsets. It is strong on tabular data because it handles nonlinear interactions and is fairly robust to heterogeneous feature distributions.

y^(x) = mode { T_b(x) } for b = 1 to B

In this project, it plays the role of the supervised known-threat detector.

One-Class SVM

One-Class SVM estimates the support of benign traffic in a high-dimensional feature space. Anything outside that learned region is treated as anomalous.

min (1/2)||w||^2 + (1/(nu n)) sum xi_i - rho
subject to: w · phi(x_i) >= rho - xi_i

In this project, it plays the role of the novelty detector for zero-day attacks.

Closed-World Strength

Random Forest is a good choice when attack labels exist and the future resembles the training distribution.

Open-World Strength

One-Class SVM is useful when malicious behavior cannot be completely enumerated ahead of time.

Evaluation Implication

Standard metrics alone are not enough. Zero-day recall matters because a model can look good overall while still failing on unseen attacks.

06 / Model Application

How the Two Evaluation Paths Were Designed

Random Forest Split Logic

  • Benign traffic is divided into train and test sets.
  • DoS attacks-Hulk is treated as the seen attack available for training.
  • Bot and DoS attacks-SlowHTTPTest are withheld as unseen attacks.
  • Zero-day recall is computed on the withheld attack subset.

One-Class SVM Logic

  • Train only on benign traffic.
  • Hold out benign samples for testing.
  • Evaluate against all attack samples at test time.
  • Treat every malicious example as zero-day from the model's point of view.

Random Forest Output

Best interpreted as the known-threat baseline. It should benefit from labeled attack structure, but its zero-day behavior is the real question.

One-Class SVM Output

Best interpreted as the novelty detector. It usually pays for broader detection sensitivity with a more difficult precision tradeoff.

What RF Tells Us

Random Forest achieved perfect precision but zero zero-day recall. That means it was extremely selective and failed to catch unseen attack categories under this evaluation split.

What OCSVM Tells Us

One-Class SVM delivered lower recall overall, but it did detect a portion of unseen attacks. That is exactly why anomaly-based modeling is relevant to zero-day security.

Main Takeaway

The better zero-day story in this project comes from model complementarity, not from a single metric win. Supervised accuracy alone is not enough.

07 / Main Findings

What This Project Demonstrates

Finding 1

The dataset requires real preprocessing. Without cleaning, missing-value handling, artifact removal, and correlation pruning, the pipeline is not reliable.

Finding 2

Zero-day framing changes the evaluation story. The recorded Random Forest run had perfect precision but zero zero-day recall, proving that a conventional classifier is not automatically strong under novelty conditions.

Finding 3

The One-Class SVM only reached 0.1445 zero-day recall, but that is still materially better than zero. The combination of supervised and anomaly-based models tells a more realistic cyber-defense story than either one alone.

09 / Limitations and Next Steps

What Still Needs to Improve

Current Limitations

  • The processed artifact in the repo reflects the validated data product already present, not the full possible raw ingestion.
  • Feature engineering is heuristic and correlation-driven rather than importance-optimized.
  • There is no consolidated metrics table saved into the repo yet.
  • Hyperparameter tuning and repeated evaluation are still limited.
  • The current Random Forest configuration does not generalize to unseen attacks in this split.

Best Next Iteration

  • Regenerate the processed dataset from the full raw archive in one controlled run.
  • Persist model metrics to CSV or JSON for direct inclusion in the report.
  • Add threshold analysis for anomaly detection.
  • Add a thin orchestration layer so preprocessing, analysis, and training are easier to reproduce.
10 / Closing

Final Takeaway

The strongest idea in this project is not any one algorithm. It is the evaluation stance: intrusion detection should be judged not only on how well it recognizes known attacks, but also on how it behaves when the attack taxonomy changes. That is why the hybrid setup matters.

In this project, the recorded results support that claim directly: the supervised model produced stronger conventional precision, while the anomaly detector produced the only non-zero zero-day recall.
